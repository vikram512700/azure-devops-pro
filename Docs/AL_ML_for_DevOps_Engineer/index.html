🤖 AI & Machine Learning Basics

What is Artificial Intelligence?

AI is the broad field of creating machines that can perform tasks requiring human-like intelligence: understanding language, recognizing images, making decisions, solving problems.

Types of AI

🎯 Narrow AI (Weak AI)

AI designed for specific tasks

  • Virtual assistants (Siri, Alexa)
  • Recommendation systems (Netflix, Amazon)
  • Spam filters
  • Image recognition
  • This is what we build in production today

🧠 General AI (Strong AI)

AI with human-level intelligence across all domains

  • Still theoretical/research stage
  • Can learn any intellectual task
  • Not relevant for current DevOps work

Machine Learning - The Core of Modern AI

Machine Learning is a subset of AI where systems learn patterns from data rather than being explicitly programmed with rules.

Traditional Programming: Rules + Data → Output

Machine Learning: Data + Desired Output → Rules (Model)

Types of Machine Learning

1. Supervised Learning

Definition: Learning from labeled data (input-output pairs)

Use Cases:

  • Email spam detection (email → spam/not spam)
  • House price prediction (features → price)
  • Medical diagnosis (symptoms → disease)
  • Sentiment analysis (text → positive/negative)

DevOps Context: Log classification, anomaly detection in metrics, ticket categorization

2. Unsupervised Learning

Definition: Finding patterns in unlabeled data

Use Cases:

  • Customer segmentation (grouping similar users)
  • Anomaly detection (finding outliers)
  • Data compression

DevOps Context: Clustering server behaviors, detecting unusual deployment patterns

3. Reinforcement Learning

Definition: Learning through trial and error with rewards/penalties

Use Cases:

  • Game playing (AlphaGo)
  • Robot control
  • Resource optimization

DevOps Context: Auto-scaling policies, resource allocation optimization

Deep Learning

Deep Learning is a specialized ML technique using artificial neural networks with multiple layers (inspired by the human brain).

Why "Deep"? Multiple layers allow learning hierarchical representations:

  • Layer 1: Edges and simple shapes
  • Layer 2: Object parts
  • Layer 3: Complete objects

🏋️ Model Training Deep Dive

What is Model Training?

Model Training is the process of feeding data to an ML algorithm so it can learn patterns and make predictions. The algorithm adjusts internal parameters (weights) to minimize prediction errors.

Training Process Overview

1. Initialize Model
Start with random weights/parameters
2. Forward Pass
Input data → Model → Predictions
3. Calculate Loss
Measure error between predictions and actual values
4. Backward Pass (Backpropagation)
Calculate gradients to update weights
5. Update Weights
Adjust parameters to reduce error
6. Repeat
Iterate through data multiple times (epochs)

Key Training Concepts

Epochs, Batches, and Iterations

Epoch: One complete pass through the entire training dataset

Batch: A subset of the training data processed together

Iteration: One update of the model weights (one batch processed)


Example: 10,000 samples, batch size 100

  • Iterations per epoch: 10,000 / 100 = 100
  • Training for 50 epochs = 5,000 total iterations

Learning Rate

Learning Rate controls how much to adjust weights during each update.

  • Too high: Model oscillates and doesn't converge
  • Too low: Training is slow, may get stuck in local minima
  • Common values: 0.001, 0.01, 0.1

Loss Functions

Loss Function Use Case Description
Mean Squared Error (MSE) Regression Average squared difference between predicted and actual values
Binary Cross-Entropy Binary Classification Measures error for two-class problems (spam/not spam)
Categorical Cross-Entropy Multi-class Classification Error for multiple classes (cat/dog/bird)

Training Example (PyTorch)

import torch import torch.nn as nn import torch.optim as optim # Define model class SimpleNet(nn.Module): def __init__(self): super().__init__() self.fc1 = nn.Linear(10, 64) self.fc2 = nn.Linear(64, 32) self.fc3 = nn.Linear(32, 1) def forward(self, x): x = torch.relu(self.fc1(x)) x = torch.relu(self.fc2(x)) return self.fc3(x) # Initialize model = SimpleNet() criterion = nn.MSELoss() optimizer = optim.Adam(model.parameters(), lr=0.001) # Training loop for epoch in range(100): for batch_data, batch_labels in train_loader: # Forward pass predictions = model(batch_data) loss = criterion(predictions, batch_labels) # Backward pass optimizer.zero_grad() loss.backward() # Update weights optimizer.step() print(f'Epoch {epoch}, Loss: {loss.item()}')

Hyperparameters

Hyperparameters are settings you configure before training (not learned from data):

  • Learning rate
  • Number of epochs
  • Batch size
  • Number of layers/neurons
  • Dropout rate
  • Regularization strength

Hyperparameter Tuning: Finding the best combination through experimentation (Grid Search, Random Search, Bayesian Optimization)

Training Infrastructure

Component Purpose Azure Options
CPU Basic models, data preprocessing Standard VMs (D-series)
GPU Deep learning, large models NC-series VMs (NVIDIA)
Memory Large datasets in RAM E-series VMs (memory optimized)

Training Metrics & Evaluation

Metric What It Measures
Accuracy % of correct predictions (good for balanced datasets)
Precision Of positive predictions, how many were correct?
Recall Of actual positives, how many did we find?
F1 Score Harmonic mean of precision and recall

Overfitting and Underfitting

⚠️ Overfitting

Model learns training data too well, including noise

  • High training accuracy, low test accuracy
  • Solutions: More data, regularization, dropout, early stopping

⚠️ Underfitting

Model too simple to capture patterns

  • Low training and test accuracy
  • Solutions: More complex model, more features, train longer

  • Compute Management: Auto-scale GPU clusters, shut down when idle
  • CI/CD for Training: Trigger retraining on new data or code changes

💬 Large Language Models (LLMs)

What are LLMs?

Large Language Models are AI models trained on massive amounts of text data to understand and generate human-like language. Examples: GPT-4, Claude, Gemini, Llama.

LLM Scale Comparison

Model Parameters Context Window Provider
GPT-4 ~1.7 trillion (estimated) 128K tokens OpenAI
Claude Sonnet 4 Undisclosed 200K tokens Anthropic
Llama 3.1 8B, 70B, 405B 128K tokens Meta (Open Source)
Gemini Pro Undisclosed 1M tokens Google

Key LLM Concepts

Tokens

Tokens are the basic units of text that LLMs process (roughly 3-4 characters or 0.75 words).

Example: "Hello, world!" = ~4 tokens ["Hello", ",", " world", "!"]

Why it matters: API costs and context limits are measured in tokens

Context Window

Context Window is the maximum amount of text (in tokens) an LLM can process at once, including both input and output.

Example: Claude Sonnet 4's 200K token window = ~150,000 words = ~500 pages

Temperature

Temperature controls randomness in outputs (0 to 1):

  • Low (0.0-0.3): More focused, deterministic, consistent
  • Medium (0.4-0.7): Balanced creativity and consistency
  • High (0.8-1.0): More creative, random, diverse

Prompt Engineering

Prompt Engineering is the art of crafting effective instructions to get desired outputs from LLMs.

Best Practices:

  • Be specific: "Write a Python function that..." instead of "Write code..."
  • Provide context: Explain the use case and constraints
  • Use examples: Show desired input-output pairs
  • Specify format: "Respond in JSON format"

Using LLMs via APIs

🔍 RAG (Retrieval-Augmented Generation)

What is RAG?

RAG combines LLMs with external knowledge retrieval. Instead of relying only on training data, the LLM retrieves relevant information from a knowledge base before generating responses.

Why RAG? LLMs don't know your company's internal data. RAG solves this by fetching current, specific information on-demand.

How RAG Works

1. User Query
"What's our deployment process?"
2. Query Embedding
Convert question to vector representation
3. Vector Search
Find most relevant documents in knowledge base
4. Retrieve Context
Fetch top K most similar documents
5. Augment Prompt
Combine retrieved context + original query
6. LLM Generation
Generate answer using provided context

Key RAG Components

Embeddings

Embeddings are numerical vector representations of text that capture semantic meaning. Similar concepts have similar vectors.

Example:

  • "dog" → [0.2, 0.8, 0.1, ...]
  • "puppy" → [0.21, 0.79, 0.12, ...] (very close to "dog")
  • "car" → [0.7, 0.1, 0.9, ...] (far from "dog")

Vector Databases

Vector DB Type Best For
Azure AI Search Managed Service Azure-native, hybrid search
Pinecone Managed SaaS Dedicated vector search
Chroma Open Source Lightweight, Python-friendly
Weaviate Open Source GraphQL API, flexible

🔌 MCP (Model Context Protocol)

What is MCP?

MCP is an open standard created by Anthropic for connecting AI assistants to external data sources and tools. It's like a universal plugin system for LLMs.

Analogy: MCP is to LLMs what USB is to computers — a standard way to connect different components.

Why MCP Matters

Problem MCP Solves:

  • LLMs can't access your databases, APIs, or internal systems
  • Each LLM provider had different integration methods
  • Developers rebuilt integrations for each AI tool

MCP Solution:

  • Standardized protocol for AI-to-tool communication
  • Write once, use with any MCP-compatible LLM
  • Pre-built servers for common data sources

MCP Architecture

MCP Host (Client)
AI application (Claude Desktop, custom apps)
MCP Protocol
JSON-RPC messages over stdio/HTTP
MCP Server
Service exposing resources and tools
Data Source
Database, API, filesystem, cloud service

MCP Core Concepts

Resources

Resources are data exposed by MCP servers (read-only).

Examples: Files, database records, API responses, log entries

Tools

Tools are functions the LLM can call to perform actions.

Examples: Execute queries, send emails, create tickets, deploy infrastructure

Popular MCP Servers

Server Purpose Use Case
server-filesystem Access local files Read project docs, code files
server-postgres Query PostgreSQL Fetch application data, metrics
server-github GitHub integration Read repos, create PRs, issues
server-slack Slack integration Send messages, read channels

⚡ Model Inference

What is Inference?

Inference is using a trained model to make predictions on new, unseen data. It's the "production use" phase after training.

Training: Data → Model (learning)

Inference: New Data + Model → Predictions

Inference vs Training

Aspect Training Inference
Purpose Learn patterns from data Make predictions
Frequency Periodic (weekly, monthly) Continuous (thousands/sec)
Compute Heavy (GPUs, hours/days) Light (CPUs, milliseconds)
Latency Not critical Critical (real-time SLAs)
Scalability Single job High concurrency needed

Types of Inference

1. Batch Inference

Process large datasets in batches

Use Cases:

  • Daily fraud detection on all transactions
  • Nightly recommendations update
  • Weekly churn prediction

2. Real-Time Inference

Serve predictions immediately

Characteristics:

  • Low latency (< 100ms)
  • High availability (99.9%+)
  • Auto-scaling for variable load

Use Cases:

  • Credit card fraud during transaction
  • Real-time product recommendations
  • Chatbots and assistants

Inference Serving Example

# Flask REST API for Model Serving from flask import Flask, request, jsonify import torch app = Flask(__name__) # Load model at startup model = torch.load('model.pth') model.eval() @app.route('/predict', methods=['POST']) def predict(): data = request.json input_tensor = torch.tensor(data['features']) # Inference with torch.no_grad(): prediction = model(input_tensor) return jsonify({ 'prediction': prediction.tolist() }) if __name__ == '__main__': app.run(host='0.0.0.0', port=5000)

🔄 MLOps (ML + DevOps)

What is MLOps?

MLOps applies DevOps principles to ML systems: automating the ML lifecycle from data preparation through model deployment and monitoring.

Goal: Productionize ML models reliably, reproducibly, and at scale.

Why MLOps is Needed

⚠️ Challenges Without MLOps

  • Models trained on laptops (not reproducible)
  • Manual deployment (slow, error-prone)
  • No versioning of datasets or models
  • Models degrade over time without monitoring
  • Can't rollback to previous versions

MLOps vs DevOps

Aspect DevOps MLOps
What's deployed Code Code + Data + Model
Testing Unit, integration + Data validation, model performance
Versioning Git for code Git + DVC + Model registry
Monitoring Uptime, errors + Model drift, data quality

MLOps Lifecycle

1. Data Engineering
Collect, clean, version data
2. Model Development
Experiment, train, track with MLflow
3. Model Validation
Test accuracy, fairness, performance
4. Model Registration
Version and store in registry
5. Model Deployment
CI/CD pipeline to production
6. Monitoring & Retraining
Track drift, trigger retraining

📦 End-to-End Project Example

Project: Fraud Detection System

Scenario: Build a real-time fraud detection system for financial transactions using Azure ML and AKS.

Your Role: DevOps engineer responsible for the entire ML infrastructure and deployment pipeline.

Project Architecture

Fraud Detection System Architecture: User Transaction ↓ Azure API Management ↓ AKS Inference Service (FastAPI) ↓ ML Model (PyTorch/ONNX) ↓ Prediction + Logging ↓ Azure Monitor + Storage Training Pipeline: Azure Data Factory → Azure Blob Storage ↓ Azure ML Training Job (GPU Cluster) ↓ Model Evaluation ↓ Azure ML Model Registry ↓ Azure DevOps Pipeline ↓ AKS Deployment (Blue-Green)
